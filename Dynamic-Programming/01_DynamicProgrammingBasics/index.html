Dynamic Programming Basics

I’ve been asked a DP problem today during the interview. It was not hard but I failed to think it clearly. I feel really really bad, so I think I need to review and sum up what I’ve learnt about DP. I’ll go over the problems I’ve met in the past few years and then find some new ones to practice.

Dynamic Programming Concepts

“The basic idea of Dynamic Programming is drawn from the intuition behind divide and conquer and is essentially the opposite of the greedy strategy: one implicitly explores the space of all possible solutions, by carefully decomposing things into a series of subproblems, and then building up correct solutions to larger and larger subproblems.”[1]

Optimal Substructure and Overlapping Subproblems are the two keys attributes for Dynamic Programming.

1. Optimal Substructure

Optimal substructure means that the optimal solution to the given problem can be obtained by the combination of optimal solutions to its subproblems. Optimal substructures are usually described recursively.

For example, the Shortest Path problem has optimal substructure property while the Longest Path problem (NP hard) does not.

In the following example, the longest path from A to D is A->B->D, but this path is not a combination of longest path from A to B and from B to D since the longest path from A to B is A->C->D->B instead of A->B and the longest path from B to D is B->A->C->D.

2. Overlapping Subproblems

For Optimal Substructure, we know that the problem’s solution can be obtained by the combination of optimal solutions to its subproblems. Overlapping Subproblems means that these subproblems must overlap, which means that solutions to some subproblems are needed again and again.

If there are no common overlapping subproblems, then Dynamic Programming is not useful. (Think about Merge Sort / Binary Search, we use DC instead of DP.)

3. Memoization and Tabulation

Since in dynamic programming, there exist Overlapping Subproblems, we could store the intermediate results of these subproblems in a table to avoid recomputing them.

There are two ways to do this:

  1. Memoization: filling up a table recursively in top-down manner.
  2. Tabulation: filling up a table iteratively in bottom-up manner.

4. An simple example: Fibonacci Numbers

If using Divide & Conquer:

This implementation is quite straightforward, but we have recomputed many same subproblems many times. The solution to the recurrence is exponential - we double the work on each recursive call.

Draw a recursion tree for fib(5), we can see that f(3) is calculated 2 times, f(2) is calculated 3 times…

So apparently we could store those intermediate results.

4.1 Memoization (Top Down)

This implementation really looks like the recursive version, we just use a table to store the computed results. Whenever we need solution to a subproblem, we first lookup the table. If it’s already there, just return it, if not, we calculate it and store it to the table.

Why Memoization is top down? Because we are looking for solutions from the root of the recursion tree to its bottom. To compute fib(5), we first look into fib(4) and fib(3), to compute fib(4), we first look into fib(3) and fib(2)… Top Down is for recursion tree, like a tree grows from a single root down to its leaves; for the table we still fill dp[2] first, then dp[3], dp[4]…

4.2 Tabulation (Bottom Up)

Tabulation is a iterative version.

Why Tabulation is bottom up? To compute fib(n), we always first compute fib(0), then fib(1), then fib(2), then fib(3) and so on. So we are building the solutions of subproblems bottom-up, like a tree grows from leaves to root (sounds like the tree is growing unnaturally)…
